# Design: `/revise-claude-md` — Rich Intermediate CLAUDE.md Skill

**Date:** 2026-06-28
**Status:** Approved

## Problem

The current intermediate CLAUDE.md generation (`renderDomainClaudeMd`) produces thin skeletons
with `AI-ENRICH` placeholders. The enrichment pass in `/init` step 5.5 fills them in, but its
instructions do not tell Claude to read `.claude/rules/`, read actual source files for concrete
patterns, or expand beyond the domains the deterministic scan detected.

Result: domain CLAUDE.md files are generic and low-signal compared to what the
`claude-md-management:revise-claude-md` plugin produces (e.g. the nestjsproject example, which
includes concrete class names, specific injection patterns, bcrypt cost factors, JWT payload
shapes, and references to specific rule files with condensed summaries).

## Goal

A standalone skill `/revise-claude-md` that generates rich, concrete intermediate CLAUDE.md files
for strategic subdirectories of any target project. It:

1. **Starts from the scan-detected domains** + expands with its own analysis
2. **Reads `.claude/rules/`** (and subdirectories — where aia-harness puts per-stack rules)
3. **Maps rules to domains** by relevance, not just by citation
4. **Reads real source files** for concrete patterns (class names, DI tokens, naming conventions)
5. **Absorbs rule content inline** (condensed 1-3 bullet summary) + references the rule file
6. **Two-phase flow** matching the harness consent-gate pattern: map → approve → generate with diffs

## Scope

- **Root CLAUDE.md**: unchanged — generated by `plan/apply` and enriched by init step 5.5 as today
- **Intermediate domain CLAUDE.md**: fully owned by this skill
- **Pipeline (`scan → plan → apply`)**: unchanged
- **`/init`**: adds step 5.6 that invokes this skill after 5.5

## Design

### New file: `skills/revise-claude-md/SKILL.md`

Standalone skill. Can be invoked:
- Manually: `/revise-claude-md [path]`
- By `/init` at step 5.6 (after root enrichment)

### Phase 1 — Map

**Step 1: Get base domains**

```bash
node "${CLAUDE_PLUGIN_ROOT}/bin/harness.mjs" scan <target> --json
```

Extract `profile.architecture.domains[]` as the starting domain list.

**Step 2: Expand domains**

Walk the target file tree. Add any directory meeting **any** of these criteria:

| Criterion | Example |
|---|---|
| Has a framework module file (`*.module.ts`, `*Module.java`, `module.go`, `router.go`) | `auth/auth.module.ts` |
| Is a recognized infrastructure layer name | `database/`, `infra/`, `repository/`, `persistence/` |
| Has ≥ 3 source files **and** is a named feature directory | `users/` with service + controller + dto |
| Canonical domain names (any path segment) | `auth`, `database`, `db`, `config`, `core`, `shared`, `common`, `gateway`, `queue`, `jobs`, `billing`, `notifications` |

Cap at **20 domains** total (same as `DOMAIN_LIMIT`). If more candidates, rank by:
1. Has module file → highest priority
2. Canonical domain name match
3. File count (desc)

**Step 3: Map rules to domains**

For each file in `.claude/rules/**/*.md` (recursive, including `ecc/`, `stack/`, and any
subdirectory aia-harness creates):

1. File name → direct heuristic mapping (e.g. `auth-security.md` → `auth/`, `security/`)
2. Read first 5 lines → extract library/framework/layer keywords
3. Cross-reference with domain candidates

Mapping table (non-exhaustive, extended by keyword scan):

```
auth-security.*     → auth/, security/
*-database.*, *-orm.*, drizzle-*, prisma-*  → database/, db/, repository/
nestjs-architecture.* → src/, apps/api/src/
*-testing.*, testing-* → all domains with spec files
typescript-*        → all TypeScript domains
*-validation.*      → all domains with dto/ subdirs
*-error-*           → all domains
*-observability.*   → all domains
api-versioning.*    → controllers, src/ level
interceptors.*      → src/ level, middleware/
swagger-*           → controller-bearing domains
ecc/**              → matched by stack/language keyword
```

Rules with no domain-specific match → cited in root CLAUDE.md only, skipped in intermediaries.

**Step 4: Present plan**

Show table before generating anything:

```
Domain               Role                        Key files found     Applicable rules
apps/api/src/auth    Authentication module       auth.service.ts     auth-security.md, testing-jest.md
                                                 jwt.strategy.ts
apps/api/src/database DB access layer            database.module.ts  drizzle-database.md, typescript-lint.md
                                                 schema/index.ts
```

Ask which domains to include (multi-select, all pre-selected).

---

### Phase 2 — Generate

For each approved domain, in sequence:

**Step 1: Read source files**

Read up to 8 key files from the domain directory, prioritized:
1. Module/entry file (`*.module.ts`, `index.ts`, `mod.go`, etc.)
2. Service file (`*.service.ts`, `*Service.java`, etc.)
3. Controller/handler
4. Schema/entity/model file
5. DTO/input file
6. Guard/middleware
7. `index.ts` or barrel
8. Test file (for convention extraction)

**Step 2: Read applicable rule files**

Read each rule file mapped to this domain in full.

**Step 3: Generate CLAUDE.md**

Structure (no `AI-ENRICH` markers — content is fully generated):

```markdown
# {domain-name} — {role in one line}

{1-2 sentences of context: what this module/layer does; note if security-sensitive}

## Responsibility

{2-4 concrete sentences: what belongs here, what does NOT, and where that other code lives}

## Key patterns

{3-6 bullets derived from real source files: specific class names, DI tokens,
naming conventions, error patterns, method names that matter}

## Applied rules

Rules active in this directory — read them when touching code here:

- @.claude/rules/{rule1}.md — {condensed 1-2 sentence summary of what matters IN THIS domain}
- @.claude/rules/{rule2}.md — {condensed summary}

## Local conventions

{2-5 directory-specific conventions derived from the real files read}

<!-- Generated by aia-harness revise-claude-md. Re-run /revise-claude-md to update. -->
```

**Step 4: Diff + consent**

Show diff vs. existing file (if any). Ask for confirmation before writing. If the file is a
previously-generated skeleton (contains `AI-ENRICH` markers), note that explicitly.

## What does NOT change

- Root `CLAUDE.md` generation and enrichment
- `scan → plan → apply` pipeline (deterministic, unchanged)
- `renderDomainClaudeMd` skeleton still generated by `apply` as fallback (updated AI-ENRICH instructions only)
- `doctor` — already detects residual `AI-ENRICH` markers and will suggest running `/revise-claude-md`
- `patch` — unchanged

## Domain CLAUDE.md quality bar (acceptance criteria)

A generated domain CLAUDE.md passes if:

1. `## Responsibility` names at least one thing that does NOT belong here (where it lives)
2. `## Key patterns` cites at least one real class name / symbol / token from the actual source files
3. `## Applied rules` has at least one `@.claude/rules/X.md` reference with a condensed summary
4. No `AI-ENRICH` markers remain
5. No generic advice repeated from the root CLAUDE.md without domain-specific context
