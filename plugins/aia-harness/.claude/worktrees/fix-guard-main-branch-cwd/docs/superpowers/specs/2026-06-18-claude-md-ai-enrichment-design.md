# Design: CLAUDE.md AI Enrichment + Stack-Aware Skills Block

**Date:** 2026-06-18  
**Status:** Approved  
**Scope:** `lib/data/skill-map.mjs` (new), `lib/generate/claude-md.mjs` (extend), `commands/init.md` (extend)

---

## Problem

`renderRootClaudeMd()` gera um CLAUDE.md puramente determinístico com:
- Mapa de arquitetura = lista plana de `domain.path → domain.role` (detectado estaticamente)
- Convenções = 4 linhas genéricas hardcoded
- Nenhuma referência a skills que o agente deve usar para a stack do projeto

O resultado é um CLAUDE.md de baixo valor: o agente ignora convenções genéricas e o mapa não descreve relacionamentos entre módulos.

---

## Decisões de design

| Questão | Decisão |
|---|---|
| Onde gera skills block | Engine (estático) — mapeamento `framework → skill` em `lib/data/skill-map.mjs` |
| Onde gera architecture map rico | Agente no `/init` — 3 passes de análise pós-apply |
| Quando enriquece | Após `apply` (step 5), antes do `review` (step 6) |
| Fallback se IA falhar | Esqueleto determinístico permanece (fail-open) |

---

## Componentes

### 1. `lib/data/skill-map.mjs` (NOVO)

Módulo puro. Exporta `skillsForProfile(profile) → Array<{label, skill, description}>`.

`skillsForProfile` opera sobre duas fontes do profile:
- `profile.frameworks[].name` — frameworks detectados (Next.js, React, Adianti, etc.)
- `profile.dependencies` — deps brutas do projeto (para ferramentas que não são frameworks: pg, prisma, amqplib, etc.)
- `profile.primaryLanguage` — para linguagens sem framework JS (Go, Java, Kotlin, etc.)

Mapeamento completo (avaliado em ordem; primeiros matches ganham — sem duplicatas):

| Fonte | Trigger | Skills injetadas |
|---|---|---|
| framework | Next.js | `vercel:nextjs` |
| framework | Nuxt | `vercel:nuxt` |
| framework | React *(somente se Next.js ausente)* | `vercel:react-best-practices` |
| framework | Angular | `ui-ux-pro-max` |
| framework | SvelteKit | `ui-ux-pro-max` |
| framework | Svelte *(somente se SvelteKit ausente)* | `ui-ux-pro-max` |
| framework | NestJS / Express / Fastify / Koa / Hono / AdonisJS | `node` |
| framework | Quarkus | `quarkus-patterns`, `quarkus-verification` |
| framework | Spring Boot | `java-coding-standards` |
| language | Java *(sem Spring/Quarkus)* | `java-coding-standards` |
| language | Go | `golang-code-style`, `golang-design-patterns`, `golang-modernize` |
| framework | Adianti | `adianti-framework` |
| dep | `@shadcn/ui` ou `shadcn` | `vercel:shadcn` |
| dep | `langchain` / `@langchain/*` | `langchain` |
| dep | `remotion` | `remotion-best-practices` |
| dep | `amqplib` / `rabbitmq-client` / `@nestjs/microservices` com RabbitMQ | `rabbitmq-development` |
| dep | `pg` / `postgres` / `@prisma/client` / `typeorm` + postgres em config | `postgresql-database-engineering` |
| fallback | primaryLanguage é JS/TS + tem framework `category:"frontend"` + nenhuma skill frontend mapeada acima | `ui-ux-pro-max` |

**Deduplicação:** Next.js presente → React não adiciona `react-best-practices`. SvelteKit presente → Svelte não adiciona skill separada. Java com Spring ou Quarkus → não adiciona `java-coding-standards` via regra de linguagem (já coberto pelo framework).

### 2. `lib/generate/claude-md.mjs` — extensões

**Nova função `skillsBlock(profile)`:**
- Chama `skillsForProfile(profile)` 
- Se resultado vazio → retorna `""` (seção omitida)
- Caso contrário, retorna bloco markdown formatado

**Mudança em `renderRootClaudeMd()`:**
- Injeta `skillsBlock(profile)` após `## Canonical commands`
- Adiciona comentários `<!-- AI-ENRICH -->` nas seções de architecture map e conventions como marcadores para o agente

Estrutura do esqueleto gerado pela engine:

```markdown
# {nome}

> Project memory for Claude Code. Keep this file short and high-signal —
> bloated memory gets ignored. Put hard guarantees in hooks, not prose.

## Stack
{stackLine}
Architecture: **{style}**{monorepo}.

## Canonical commands
Always use these exact commands (do not guess):

{commandsBlock}

## Skills — use para esta stack

> Invoque a skill correspondente antes de trabalhar no domínio dela.

- **{Framework}** → `/{skill}` — {description}

## Architecture map
<!-- AI-ENRICH: analyze file tree and key source dirs, describe module responsibilities and relationships, replace this section -->

{domainMap}

## Conventions
<!-- AI-ENRICH: detect project-specific patterns from source files, replace with 4-7 concrete conventions, remove generic ones -->

- Match the style of surrounding code; do not introduce new patterns unprompted.
- Write unit tests for every new function or module added.
- Run the lint + test commands above before claiming work is complete.
- Never commit secrets; environment values belong in `.claude/settings.local.json` (gitignored).

<!-- Generated by aia-harness. Edit freely; re-run /aia-harness:doctor to audit. -->
```

### 3. `commands/init.md` — step 7.5 (NOVO)

Inserido entre apply (step 5) e review (step 6). Obrigatório — não é "offer to tailor".

#### Step 7.5 — Enrich CLAUDE.md (3 passes)

**Pass 1 — Estrutura e responsabilidades**

```bash
find {target} -maxdepth 3 \
  -not -path '*/node_modules/*' \
  -not -path '*/.git/*' \
  -not -path '*/vendor/*' \
  -not -path '*/.next/*' \
  -not -path '*/dist/*' \
  -not -path '*/.nuxt/*' \
  -not -path '*/target/*'
```

Leia também:
- `README.md` ou `README` (raiz)
- `docs/` até 2 níveis (se existir)
- Arquivos de config na raiz: `package.json`, `composer.json`, `pyproject.toml`, `pom.xml`, `go.mod`, `Cargo.toml`

Objetivo: entender responsabilidades dos módulos top-level e relações entre eles.

**Pass 2 — Padrões reais do código fonte**

Com base nos frameworks detectados no profile, leia os diretórios-chave abaixo. Leia 2–4 arquivos representativos por diretório.

| Framework | Diretórios a analisar |
|---|---|
| Next.js | `app/`, `src/app/`, `pages/`, `middleware.ts`, `lib/` |
| Nuxt | `pages/`, `server/`, `composables/`, `plugins/` |
| React (SPA) | `src/`, `src/components/`, `src/hooks/`, `src/pages/` |
| Angular | `src/app/`, `src/app/core/`, `src/app/shared/` |
| Laravel | `app/Http/Controllers/`, `routes/`, `app/Models/`, `resources/views/` |
| Adianti | `app/control/`, `app/model/`, `app/view/`, `index.php` |
| NestJS | `src/`, `src/modules/`, `src/common/` |
| Quarkus | `src/main/java/`, `src/main/resources/` |
| Spring Boot | `src/main/java/`, `src/main/resources/application*.yml` |
| Go | `cmd/`, `internal/`, `pkg/`, `api/` |
| Django | `apps/`, `config/`, `requirements*.txt` |
| FastAPI | `app/`, `app/routers/`, `app/models/` |

Objetivo: detectar padrões reais — naming conventions, estrutura de imports, error handling, padrões de teste, organização de módulos.

**Pass 3 — Síntese e escrita**

Com os dados dos passes 1 e 2, reescreva o CLAUDE.md do projeto alvo:

1. Reescreva `## Architecture map`:
   - Uma linha por módulo/diretório relevante
   - Descreva **responsabilidade + relacionamentos** (ex: "protegido por middleware X", "consome serviço Y via lib/Z")
   - Omita diretórios óbvios (`node_modules`, `dist`, `.git`)
   - Máximo 15 entradas

2. Reescreva `## Conventions`:
   - 4–7 convenções **específicas ao projeto** detectadas no código-fonte
   - Concretas e acionáveis (ex: "Server Components por padrão; `use client` apenas para interatividade")
   - Sem convenções genéricas que se aplicariam a qualquer projeto

3. Remova os comentários `<!-- AI-ENRICH: ... -->` do arquivo final

4. **Não altere** as seções `## Stack`, `## Canonical commands`, `## Skills`

5. Mostre diff entre esqueleto e versão enriquecida → aguarde aprovação explícita → aplique com `Edit`

---

## Invariantes preservadas

- Engine (`lib/`) permanece pura e sem IO adicional — `skill-map.mjs` é módulo de dados puro
- Fail-open: se agente não enriquecer, esqueleto com fallback estático permanece funcional
- Comentários `AI-ENRICH` são removidos no passo final — não persistem no arquivo do usuário
- Seções determinísticas (stack, commands, skills) nunca alteradas pelo agente
- `skillsBlock` omitido se nenhuma skill mapeada → sem ruído para stacks não cobertas
